# 자동 코드 리뷰 (auto-code-review)

이 스킬은 코드를 체계적으로 리뷰해 **정확성 버그**, **보안 취약점**, **개선 가능한 부분**을 찾아낸다.
호출되면 아래 절차를 그대로 따른다. 사용자가 별도로 범위를 지정하지 않으면 스스로 리뷰 대상을 판단한다.

## 1. 리뷰 대상 결정

우선순위대로 확인해 리뷰할 코드를 정한다.

1. **사용자가 특정 파일/함수/코드 조각을 지정** → 그것만 리뷰.
2. **git 저장소인 경우** → 커밋되지 않은 변경분과 최근 커밋을 대상으로:
   - `git status` 로 변경 파일 확인
   - `git diff` (working tree) 와 `git diff --staged` 로 변경 내용 확보
   - 변경이 없으면 `git diff HEAD~1` 로 마지막 커밋을 리뷰
3. **git 저장소가 아니거나 방금 이 세션에서 코드를 작성한 경우** → 이번 대화에서 생성·수정한 파일을 대상으로.
4. **위 어느 것도 없으면** → 사용자에게 "어떤 파일 또는 디렉터리를 리뷰할까요?"라고 물은 뒤 진행.

리뷰 전, 대상 파일을 Read로 실제로 읽어 전체 맥락을 파악한다. diff만 보고 판단하지 않는다. 주변 코드·호출부·타입 정의를 함께 확인해야 오탐을 줄일 수 있다.

## 2. 점검 항목

각 대상에 대해 아래 카테고리를 순서대로 점검한다.

### 🐞 정확성 (Correctness) — 최우선
- 경계 조건: off-by-one, 빈 배열/컬렉션, null·undefined, 0·음수 입력
- 예외/에러 처리 누락, 삼켜지는 에러
- async/await 누락, race condition, 정리되지 않는 리소스(파일·소켓·타이머)
- 잘못된 비교(`==` vs `===`), 타입 강제 변환 함정
- 로직이 의도한 결과를 내는가 — 구체적 입력을 대입해 반례를 찾는다

### 🔒 보안 (Security)
- 입력 검증 없는 사용자 데이터 → 인젝션(SQL/명령/XSS)
- 하드코딩된 시크릿·토큰·비밀번호
- 안전하지 않은 역직렬화, 경로 조작(path traversal)
- 인증·권한 확인 누락

### ⚡ 성능·효율 (Efficiency)
- 불필요한 반복/중복 연산, O(n²) 이상으로 쉽게 개선 가능한 부분
- 루프 내 반복 I/O·쿼리(N+1)
- 불필요한 리렌더/재계산

### 🧹 재사용·단순화 (Cleanup)
- 중복 코드 → 기존 유틸/함수로 대체 가능한지
- 과도하게 복잡한 분기, 죽은 코드, 미사용 변수
- 프로젝트의 기존 네이밍·패턴과 어긋나는 부분

## 3. 검증 (오탐 제거)

발견 항목을 보고하기 전에 각각을 **실제로 문제가 되는지** 스스로 검증한다.
- "이 입력이 들어오면 이 줄에서 이런 잘못된 결과/크래시가 난다"를 구체적 시나리오로 말할 수 있어야 한다.
- 시나리오를 구성할 수 없으면 그 항목은 버리거나 확신도를 낮춰 표기한다.
- 스타일 취향이나 사소한 선호는 지적하지 않는다. 실질적 영향이 있는 것만 남긴다.

## 4. 결과 보고

발견을 **심각도 높은 순**으로 정렬해 아래 형식으로 출력한다.

```
## 코드 리뷰 결과 — <대상>

### 심각 (Critical) 🔴
- **[파일:줄]** 한 줄 요약
  - 문제: 무엇이 왜 잘못되는가
  - 재현: 어떤 입력/상황에서 터지는가
  - 제안: 어떻게 고치면 되는가 (필요 시 코드 예시)

### 경고 (Warning) 🟡
- ...

### 제안 (Suggestion) 🟢
- ...
```

- 발견이 **하나도 없으면**, 억지로 만들지 말고 "확인한 범위에서 실질적 문제를 찾지 못했습니다"라고 명확히 밝힌다. 무엇을 점검했는지 한 줄로 요약한다.
- 각 발견에는 반드시 `파일:줄`을 붙여 클릭 가능한 위치를 제공한다.

## 5. (선택) 수정 적용

사용자가 "고쳐줘" 또는 "--fix"를 함께 요청한 경우에만, 보고한 발견을 실제 코드에 반영한다.
- 심각·경고 항목부터 적용하고, 각 수정이 무엇을 바꿨는지 요약한다.
- 요청하지 않았다면 **리뷰만 하고 코드를 수정하지 않는다.**

## 원칙

- **정확성 > 보안 > 성능 > 정리** 순으로 우선한다. 사소한 지적으로 목록을 늘리지 않는다.
- 확신 없는 지적에는 "확인 필요"라고 명시해 사용자가 판단하게 한다.
- 리뷰 대상 코드를 임의로 실행하거나 외부로 전송하지 않는다.
